## Verified already correct — no change needed

- **`docs/CROSSWALK_GUIDE.md`** line 134: "CMMC 2.0 Level 2 = NIST 800-171
  (110 identical practices). If you have 800-171, you're aligned with CMMC
  Level 2." — this was already accurate about the *official standards'*
  relationship before this PR; it's now also accurate about what
  ControlWeave's seeded `cmmc_2.0` control set actually contains (previously
  only 54 of the 110 were seeded, so the practical crosswalk coverage this
  line implies didn't fully exist yet). Same "previously-true-of-the-standard,
  now-also-true-of-the-seed" pattern noted for `docs/frameworks/NIST_800_53.md`
  in PR #576's review.

  implementations | ~110 unique | ~50%" — describes the crosswalk dedup
  math assuming both frameworks are fully seeded at 110 each; `nist_800_171`

  PR description), so this line's *practical* accuracy still depends on that
  separate, not-yet-done completion. Noted below, not fixed here.

  describes the official standard's three maturity levels and practice
  counts (Level 2 = 110 practices), not what ControlWeave seeds. Accurate
  as written; this file already carries a stale-notice banner (from the
  separate framework-catalog-scope docs work) pointing readers to the live

- **`docs/guides/FAQ.md`**, **`docs/guides/RMF.md`**,

  Still accurate.

## Noted, not fixed (out of scope for this PR)

- `nist_800_171` is still a 24-control subset (labeled "Rev 3" in the seed,
  though the practices it does have use Rev-3-style zero-padded IDs like
  `03.01.01`). Completing it properly needs a genuinely reachable Rev 3
  source; the source this PR uses (FATHOM5CORP/oscal) is Rev 2 only — the
  right source for CMMC, not for a `nist_800_171` "Rev 3" framework. Until
  that's done, `docs/CROSSWALK_GUIDE.md` line 90's "~110 unique" dedup math
  for the NIST 800-171 + CMMC 2.0 pairing remains aspirational rather than
  fully realized in the seed data — a pre-existing gap this PR doesn't
  create and doesn't fully close.
