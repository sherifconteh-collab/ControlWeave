# Documentation review — PR #594

Performed live by a Claude Code session with direct PR/repo access, triggered
by `docs-pipeline.yml`'s "📚 Documentation Update Queued" comment, which
flagged `cmdb` and `settings` as affected. A later commit on this PR ported
PR #591's `help.js` fix (needed to get CI's now-enforced E2E suite green —
see commit history), which triggered a second flag for `help-center`;
covered in its own section below.

Scope: `controlweave/docs/guides/CMDB.md`, `controlweave/docs/USER_GUIDE.md`,
`controlweave/docs/TIER_COMPARISON.md`, `controlweave/docs/guides/SETTINGS.md`,
`controlweave/docs/guides/GETTING_STARTED.md`, `controlweave/docs/guides/AI_ANALYSIS.md`,
`controlweave/docs/guides/AI_COPILOT.md` — reviewed against this PR's actual
code changes: `cmdb.js` (added `requireFields` validation to
environment/service-account creation), `orgSettings.js` (fixed the account
data export route's broken column names and missing email decryption),
`policies.js` and `externalAi.js` (unrelated 500-fixes, not covered by any of
the flagged docs), `help.js` (removed a leftover 403 tier-gate on
`GET /help/:slug`, ported verbatim from PR #591), plus the QA-script
rewrites (not user-facing, no doc surface).

## Verified already correct — no change needed

- **`docs/guides/CMDB.md`** — Section 2.1 ("Add an Environment") already
  lists **Name** and **Code** as "Required Fields," and Section 6.1 ("Add a
  Service Account") already lists **Account Name** as required. Neither
  route actually enforced that before this PR — a request missing `code` or
  `account_name` hit an unhandled `NOT NULL` constraint violation and
  returned a raw 500 instead of a validation error. The doc was correct
  about what the UI form requires; this PR's `requireFields` addition makes
  the API actually match it, rather than the doc needing to change.
- **`docs/guides/SETTINGS.md`** — the Account → Data Export section already
  correctly lists the six export sections (organization profile, activated
  frameworks, controls/implementations, assets, users, audit logs) that the
  export endpoint returns. The endpoint itself was fully broken (500, wrong
  column names in `orgSettings.js`'s SQL) before this PR; the fix restores
  behavior the doc already described, no wording changes needed. The doc
  makes no claim about email formatting, so the accompanying fix (returning
  the user's decrypted email instead of raw AES-GCM ciphertext) is also not
  a doc-visible change.
- **`docs/TIER_COMPARISON.md`** — already states plainly (line 3): "As of
  v4.0, all tier gating was removed — every feature listed in this product
  is available to every authenticated user... `requireTier()`/
  `requireProEdition()` middleware in the backend are no-ops." This PR's QA
  fixes (state-ai-laws no longer expected to be govcloud-gated, seat/
  framework-count limits confirmed removed) are all consistent with this
  statement — nothing to correct.
- `docs/USER_GUIDE.md`, `docs/guides/GETTING_STARTED.md` — spot-checked;
  neither describes environment/service-account required fields, the
  account-export endpoint, or seat limits in enough detail to be affected by
  this PR's changes.
